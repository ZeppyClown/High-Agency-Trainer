# Project Scope: Companion — A Silent Accountability App

## What This App Does
A silent Mac desktop app (Electron) that runs in the background like Discord.
It passively tracks what the user is doing on their computer, knows their long-term goal,
and nudges them at the right moment with one small concrete action — no shaming, no dashboards, just a gentle push.

## Who This Is For
One user. The developer himself. Goal: become a better programmer and get an internship.
That means: doing leetcode regularly + building side projects.

## Tech Stack
- Electron (desktop app shell)
- Node.js (backend logic)
- React (UI panels)
- SQLite via better-sqlite3 (local data storage)
- Groq Cloud API (nudge generation + classification — free tier)
  - Primary: llama-3.3-70b (smarter, for nudge generation)
  - Fallback: llama-3.1-8b (faster, for classification)
  - Final fallback: local rules (app never breaks if API is down)
- macOS APIs via AppleScript / active-win npm package (window tracking)
- node-notifier or Electron Notification API (nudges)

## Developer Profile
- Intermediate JS, can build web apps
- No prior Electron or desktop experience
- Mac only, 1-2 hours per day
- Learning mode: AI explains concepts and guides — does NOT write the code for the user

## Core Features (V1 only — nothing else)

### 1. Passive Window Tracker
- Every 10 seconds, log the currently active application and window title
- Store: { timestamp, appName, windowTitle }
- Categorise using a two-layer hybrid system:
  - Layer 1 (local, instant): obvious rules — vscode/cursor/zed → "coding", known game executables → "gaming"
  - Layer 2 (AI batch, every 30 mins): send URL + window title to Groq with user's goal as context, let AI classify ambiguous cases
  - DO NOT fetch page HTML — URL + title is sufficient signal and keeps token usage tiny (~50 tokens per batch)
- Fallback chain: llama-3.3-70b → llama-3.1-8b → local rules if API is down

### 2. Goal Setup (one-time)
- Simple onboarding screen on first launch
- User types their goal in plain English: "I want to be a better programmer and get an internship"
- App infers from this: required habits = leetcode + coding projects
- Stored locally in SQLite

### 3. Nudge Engine
- Runs a check every 30 minutes silently
- Conditions that trigger a nudge:
  - User has been in "distraction" category for 60+ continuous minutes
  - User has not done leetcode in 3+ days
  - User has not coded in 2+ days
- When triggered: show a Mac notification with one concrete action
  - Example: "You haven't leetcoded in 3 days. Here's an easy problem to start: [link]"
  - Example: "2 hours on YouTube. Your project is sitting there — open VSCode for 20 mins?"
- Nudge is generated by AI using the context (goal + recent activity summary)
- Maximum 2 nudges per day — never spammy

### 4. Weekly Summary (end of week, Sunday 9pm)
- Simple notification that opens a small window
- Shows:
  - Days coded this week
  - Days leetcoded this week
  - Biggest distraction app
  - One sentence from AI: "Good week" or "You can push more — here's where the time went"

## Day-by-Day Learning Plan
Assume 1-2 hours per day. Each day has one focused concept and one working output.

### Week 1 — Foundation: Electron + Window Tracking

**Day 1 — What is Electron?**
- Concept: Understand what Electron is and how it differs from a web app
- Task: Scaffold a basic Electron app that opens a window and shows "Hello Companion"
- Output: App launches on your Mac
- Key things to learn: main process vs renderer process, package.json scripts

**Day 2 — System Tray App**
- Concept: How to make an app that lives in the menu bar, not as a window
- Task: Move your app to the system tray with a basic right-click menu (Quit, Open)
- Output: App sits silently in Mac menu bar
- Key things to learn: Tray API in Electron

**Day 3 — Active Window Tracking**
- Concept: How to read the currently active macOS window from Node.js
- Task: Install active-win, poll every 10 seconds, console.log the app name and title
- Output: Terminal prints what app you're using every 10 seconds
- Key things to learn: npm packages in Electron, setInterval in main process

**Day 4 — Categorisation Logic**
- Concept: How to write a simple rules-based classifier in JS
- Task: Write a categorise(appName, windowTitle) function that returns "coding", "leetcode", "distraction", or "other"
- Output: Console shows category alongside app name
- Key things to learn: switch statements, string matching, writing pure functions

**Day 5 — SQLite Setup**
- Concept: What is a local database and why use it instead of a JSON file
- Task: Install better-sqlite3, create a database, create an activity_log table, insert a test row
- Output: A .db file exists on your machine with one row of data
- Key things to learn: SQLite basics, CREATE TABLE, INSERT

**Day 6 — Logging to Database**
- Concept: Connecting the tracker to the database
- Task: Every 10 seconds, instead of console.log, write the activity to SQLite
- Output: After 5 minutes of use, you can open the DB and see a real log of what you did
- Key things to learn: calling DB functions from tracker, timestamps

**Day 7 — Rest + Review**
- Review your code, clean it up, make sure it actually runs reliably
- Try running it for a full hour and check the DB
- Fix any bugs

---

### Week 2 — Goal Setup + Onboarding UI

**Day 8 — Add React to Electron**
- Concept: How to use React inside an Electron renderer
- Task: Set up a basic React component that renders inside your Electron window
- Output: Electron window shows a React UI
- Key things to learn: Electron + React setup, renderer process

**Day 9 — Onboarding Screen**
- Concept: Building a simple form UI in React
- Task: Build a screen where user types their goal and clicks Save
- Output: A clean input screen with a button
- Key things to learn: useState, controlled inputs, basic Tailwind or CSS

**Day 10 — Save Goal to SQLite**
- Concept: Sending data from React (renderer) to Node (main process)
- Task: When user clicks Save, store the goal in a settings table in SQLite
- Output: Goal persists after app restart
- Key things to learn: ipcRenderer and ipcMain (Electron's communication bridge)

**Day 11 — First Launch Detection**
- Concept: How to show onboarding only once
- Task: Check if goal exists in DB on startup — if yes skip onboarding, if no show it
- Output: App behaves correctly on first vs subsequent launches
- Key things to learn: conditional logic in main process, app lifecycle

**Day 12 — Query Activity Data**
- Concept: Reading and aggregating data from SQLite
- Task: Write a function that returns: total minutes per category for today
- Output: Console shows "coding: 45min, distraction: 90min, leetcode: 0min"
- Key things to learn: SELECT, GROUP BY, date filtering in SQLite

**Day 13 — Display Today's Summary in UI**
- Task: Show today's category breakdown in a simple React UI panel
- Output: You can open the app and see what you've done today
- Key things to learn: fetching data from main process to renderer via IPC

**Day 14 — Rest + Review**
- Use the app for real for two days, note what feels broken or missing
- Refactor anything messy

---

### Week 3 — The Nudge Engine

**Day 15 — Nudge Logic (No AI yet)**
- Concept: Writing the rule-based conditions for when to nudge
- Task: Write a shouldNudge() function that checks:
  - Has user been in distraction 60+ min continuously?
  - Has user not done leetcode in 3 days?
- Output: Function returns { shouldNudge: true, reason: "no leetcode in 3 days" }
- Key things to learn: querying time ranges in SQLite, date arithmetic in JS

**Day 16 — Mac Notifications**
- Concept: How to send a native Mac notification from Electron
- Task: When shouldNudge() returns true, fire a Mac notification with a hardcoded message
- Output: A real Mac notification appears
- Key things to learn: Electron's Notification API

**Day 17 — Connect to Groq API**
- Concept: How to call an external API from Node.js
- Task: Send the user's goal + today's activity summary to Groq (llama-3.3-70b), get back one nudge message
- Output: Console shows an AI-generated nudge string
- Key things to learn: fetch in Node, API keys, .env files, prompt design, Groq's OpenAI-compatible API format

**Day 18 — AI-Powered Nudge Notification**
- Task: Replace hardcoded nudge message with the AI-generated one
- Output: Real Mac notification with a personalised, context-aware nudge
- Key things to learn: async/await, chaining functions together

**Day 19 — Nudge Throttling**
- Task: Track when the last nudge was sent, enforce max 2 nudges per day
- Output: App never spams you
- Key things to learn: storing state in SQLite, time comparison logic

**Day 20 — Leetcode Link in Nudge**
- Task: When reason is "no leetcode", append a link to a random easy Leetcode problem in the notification
- Output: Notification includes a clickable link
- Key things to learn: Electron notification actions, opening URLs

**Day 21 — Rest + Review**
- Run the full app for 2 days straight
- Does the nudge fire at the right time? Does it feel right or annoying?

---

### Week 4 — Weekly Summary + Polish

**Day 22 — Weekly Stats Query**
- Task: Write a function that returns this week's: days coded, days leetcoded, top distraction app
- Output: Data is correct when you console.log it

**Day 23 — Weekly Summary Notification**
- Task: Every Sunday at 9pm, fire a notification that opens a summary window
- Output: Summary window appears on Sunday night
- Key things to learn: Electron scheduled tasks, cron-like timing

**Day 24 — AI Weekly Summary Sentence**
- Task: Send the week's stats to AI, get back one honest sentence
- Output: "Good week — you coded 4 days and leetcoded twice. Keep it up."

**Day 25 — App Launch on Startup**
- Task: Make Companion auto-launch when Mac starts up
- Output: App is always running silently in background without manual launch
- Key things to learn: Electron auto-launch

**Day 26 — Basic Error Handling**
- Task: Make sure the app doesn't crash if active-win fails, or API is down
- Output: App keeps running gracefully even when things go wrong
- Key things to learn: try/catch, fallbacks

**Day 27 — Final Polish**
- Clean up UI, make sure onboarding feels nice
- Make sure tray icon looks good
- Write a simple README

**Day 28 — You're Done**
- Run it for a real week
- This is now a live project on your resume
- It's also the thing that will actually help you become the programmer you want to be

## Notes for the AI Coding Agent

- Student is learning, not shipping. Explain the WHY before the HOW every single time.
- Never write full implementations. Give hints, pseudocode, and ask the student to attempt first.
- If student is stuck after a genuine attempt, walk through it step by step — still don't just paste code.
- Each day should feel completable in 1-2 hours. If a task is too big, split it.
- Encourage the student when they complete a day. This project is personal to them.
- The student is building this for themselves. Keep reminding them of that when they lose motivation.
